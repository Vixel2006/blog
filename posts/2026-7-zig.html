Why I Fell in Love with Zig

Last couple of weeks, I tried to learn Zig, and I went into it with the assumption that it's another highly-bloated piece of software with too many abstractions that make the language more complex instead of making it easier to develop with. The last language that I had experienced this with is Rust. I think the ownership approach to manage memory is actually very clever, but it's not that great for me to pair with all this unnecessary complexity that Rust introduces while working. Basically, Rust is not fun to write and read (at least for me). Some people will say that Rust is more fun to write than Go as Go is flat and stuff, and will basically tell you to go work on real problems bro, the fun thinking should be done about the problem on hand and the solution for it, not for what language concept to use in order to implement the logic you have already designed.

So what is great about Zig other than its simplicity?

comptime

comptime is actually very great. It's definitely a better version of macros in C/C++, and proc_macros in Rust. It's so fucking intuitive and easy to use. It basically comes naturally with the program flow.

C Approach: Preprocessor Magic

#include <stdio.h>
#include <stdbool.h>

// The "Macro Template"
#define DECLARE_STACK(Type, Name, Capacity) \
    typedef struct { \
        Type data[Capacity]; \
        size_t top; \
    } Name; \
    \
    static inline void Name##_push(Name* s, Type item) { \
        if (s->top < Capacity) { \
            s->data[s->top++] = item; \
        } \
    } \
    \
    static inline Type Name##_pop(Name* s) { \
        return s->data[--s->top]; \
    }

// Generating a specific type: IntStack
DECLARE_STACK(int, IntStack, 10)

int main() {
    IntStack stack = { .top = 0 };
    IntStack_push(&stack, 42);
    printf("%d\n", IntStack_pop(&stack));
    return 0;
}

Rust Approach: Procedural Macros

extern crate proc_macro;
use proc_macro::TokenStream;
use quote::quote;
use syn::{parse_macro_input, Expr, Type, parse::Parse, parse::ParseStream, Token};

// A custom parser struct to handle: Stack!(i32, 10)
struct StackArgs {
    ty: Type,
    _comma: Token![,],
    cap: Expr,
}

impl Parse for StackArgs {
    fn parse(input: ParseStream) -> syn::Result<Self> {
        Ok(StackArgs {
            ty: input.parse()?,
            _comma: input.parse()?,
            cap: input.parse()?,
        })
    }
}

#[proc_macro]
pub fn make_stack(input: TokenStream) -> TokenStream {
    let StackArgs { ty, cap, .. } = parse_macro_input!(input as StackArgs);

    let expanded = quote! {
        struct Stack {
            data: [#ty; #cap],
            top: usize,
        }

        impl Stack {
            fn new() -> Self { Self { data: [0; #cap], top: 0 } } // Simplification
            fn push(&mut self, item: #ty) {
                if self.top < #cap {
                    self.data[self.top] = item;
                    self.top += 1;
                }
            }
            fn pop(&mut self) -> #ty {
                self.top -= 1;
                self.data[self.top]
            }
        }
    };
    TokenStream::from(expanded)
}

// Generates the struct and implementation at compile time
make_stack!(i32, 10); 

fn main() {
    let mut s = Stack::new();
    s.push(100);
    println!("{}", s.pop());
}

Zig Approach: comptime

const std = @import("std");

// Just a normal function, but it returns a `type` 
// and takes compile-time arguments.
fn Stack(comptime T: type, comptime capacity: usize) type {
    return struct {
        data: [capacity]T = undefined,
        top: usize = 0,

        const Self = @this();

        pub fn push(self: *Self, item: T) void {
            if (self.top < capacity) {
                self.data[self.top] = item;
                self.top += 1;
            }
        }

        pub fn pop(self: *Self) T {
            self.top -= 1;
            return self.data[self.top];
        }
    };
}

pub fn main() !void {
    // Instantiating the type naturally
    var my_stack = Stack(i32, 10){};

    my_stack.push(1337);
    std.debug.print("{d}\n", .{my_stack.pop()});
}

I think if you have any kind of taste for programming you will clearly see that the comptime solution is much more elegant.

How Zig proved that adding new abstraction layers is not the solution for simpler software

It's very curious for me that Rust and C++ tried to fix C by adding new abstraction layers over each other and still managed to be so complicated and hard to write and read, while Zig removed the memory allocation and I/O abstractions in C (which are some of the rare abstractions in C) and have a very simple and fun language to write and read.

I think by now it's obvious that I'm a big fan of the no-hidden allocation, control-flow, or preprocessor. It's actually a very interesting philosophy to adopt, and it made the code actually great. I mean, not having hidden allocations or control-flow makes the code you write very predictable and makes debugging much easier.

Explicit Memory Allocation

const std = @import("std");

pub fn main() !void {
    // 1. Explicit Memory Strategy
    // We explicitly choose a General Purpose Allocator. The runtime does not provide one.
    var gpa = std.heap.GeneralPurposeAllocator(.{}){};
    defer _ = gpa.deinit(); // Enforce leak checking at scope exit
    const allocator = gpa.allocator();

    // 2. Explicit Dependency Injection
    // The ArrayList cannot exist in a vacuum; it *must* hold a reference to our allocator.
    var list = std.ArrayList(i32).init(allocator);
    defer list.deinit(); // Explicit cleanup

    // 3. Explicit Control Flow and Error Handling
    // Appending can fail if the system runs out of memory. 
    // Zig forces you to acknowledge this with the `try` keyword. 
    // There are no invisible exceptions catching this—it explicitly bubbles up the call stack.
    try list.append(42);
    try list.append(1337);

    std.debug.print("Elements: {any}\n", .{list.items});
}

Just having explicitly defining the allocator with the allocation strategy we want to use actually makes memory management much easier to work with and actually the defer keywords also helps a lot.

Explicit I/O Subsystem

const std = @import("std");

pub fn main() !void {
    // 1. Explicitly acquire a handle to Standard Output
    // This doesn't happen automatically; you must ask the OS subsystem for it.
    const stdout_file = std.io.getStdOut();

    // 2. Initialize a unbuffered or buffered Writer stream
    // Zig separates the raw file descriptor from the stream interface.
    // If you want buffering to prevent frequent system calls, you wrap it explicitly.
    var bw = std.io.bufferedWriter(stdout_file.writer());
    const stdout = bw.writer();

    // 3. Perform the I/O operation
    // Because I/O can always fail (e.g., broken pipe, disk full), 
    // the compiler forces you to handle the error with 'try'.
    try stdout.print("Hello, {s}!\n", .{"systems engineering"});

    // 4. Explicitly flush the buffer to the OS
    // With buffered I/O, you control exactly when the syscall happens.
    try bw.flush(); 
}

Build Systems: what got me into zig in the first place

This is actually the reason why I got into Zig in the first place. I love C so much, but if there is a thing in C that drives me crazy every time I write code in it, it is its build system. You should use CMake, Makefiles for it, you have to install, build, and link your third-party deps to your code manually. (Although I wouldn't lie, a big part of my programming skills came from the fact that I was building my C projects with no external deps because I was too lazy to do this linking process). But now when you want to create some real cool and useful stuff it's actually very important to have a good build system, and Zig does this for you, and it also gives you a good build system for C and C++.

I didn't use it with the C and C++ build system, even though I imported some C libraries in it like cuda and vulkan and it was very smooth, and for my latest project glu I have exported my Zig API functions and types into C and it was actually a great experience

Will I use zig for large projects

Absolutely. Actually I'm currently working on a robotics communication framework from scratch. I'm building it to be a very lightweight and clever replacement for ROS2 if you really want a performant solution for your robots without the ROS2 bloat, as I think ROS2 is very hated between indies, and it's not a good choice for large labs that need to work on performant stuff. I'm trying to make glu the open-source solution for this, and I'm building it with Zig. Will the backward compatibility problem bite me in the ass? Probably yes, but I think this language is good enough to risk it, and I hope glu becomes a reliable system for real production.
